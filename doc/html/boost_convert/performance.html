Converters Compared

The performance of Boost.Convert depends entirely on the performance of the converter deployed. A few converters have been tested for string conversions to the int, long int, double builtin types and to a user-defined type.

In turn, the relative and absolute performance of every particular converter depends on the platform, the compiler used and the particular implementation of the underlying conversion component (std::stream, printf, Boost.Spirit, etc.). Consequently, the results below are only a rough guide of relative performance of the mentioned converters on the tested platforms.

Compiled with gcc-4.8.2 on 32-bit Ubuntu 14.04 tests produced the following results: 

int-to-str: spirit/itostr/lcast/prntf/stream=  16.76/  15.52/  20.53/  26.80/  29.06 seconds.
lng-to-str: spirit/ltostr/lcast/prntf/stream=  16.96/  14.95/  20.65/  26.82/  28.62 seconds.
dbl-to-str: spirit/dtostr/lcast/prntf/stream= 105.56/  36.46/ 119.55/  92.42/ 109.97 seconds.
str-to-int: spirit/strtoi/lcast/scanf/stream=   2.07/   5.48/  10.66/  19.49/  20.90 seconds.
str-to-lng: spirit/strtol/lcast/scanf/stream=   2.09/   6.79/  10.21/  19.57/  20.58 seconds.
str-to-dbl: spirit/strtod/lcast/scanf/stream=   4.83/  14.65/  29.46/  33.67/  44.87 seconds.

Compiled with gcc-4.6.3 on 64-bit Ubuntu 12.04 tests produced the following results (adjusted): 

int-to-str: spirit/itostr/lcast/prntf/stream=  16.77/  14.94/  20.22/  44.24/  30.99 seconds.
lng-to-str: spirit/ltostr/lcast/prntf/stream=  18.02/  16.16/  20.74/  32.71/  30.59 seconds.
dbl-to-str: spirit/dtostr/lcast/prntf/stream=  59.88/  31.23/ 184.14/ 159.21/ 138.44 seconds.
str-to-int: spirit/strtoi/lcast/scanf/stream=   2.12/   9.84/  11.59/  27.82/  29.84 seconds.
str-to-lng: spirit/strtol/lcast/scanf/stream=   2.12/   9.55/  14.47/  27.50/  29.97 seconds.
str-to-dbl: spirit/strtod/lcast/scanf/stream=   3.45/  15.80/  17.28/  42.38/  56.25 seconds.

Based on the results we might conclude that

  • the Spirit.Qi-based converter was the fastest for string to int, long and double conversions. So, it might be a good candidate for the tasks predominantly doing that kind of conversions; Spirit.Karma to-string performance was relatively modest;
  • the std::stream-based and the printf/scanf-based converters were consistently the slowest. Still, given their maturity and availability, they might be options to consider if conversion performance is not your primary concern;
  • the extended strtol-based converter was reasonably fast and with basic formatting support might be an attractive all-rounder. It should be noted that it is nowhere as mature as boost::cnv::lexical_cast and boost::cnv::stream. So, bugs are to be expected.

For user-defined types two converters -- boost::cnv::lexical_cast and std::cnv::cstream -- were tested with the following results: 

str-to-user-type: lcast/stream=3.41/1.74 seconds.
user-type-to-str: lcast/stream=6.08/1.51 seconds.

Both converters provide string-to-type and type-to-string conversions; both deploy the same std::stream conversion engine; both provide extendible conversion framework where new types can be plugged in to the framework via: 

std::istream& operator>>(std::istream&, TypeOut&);
std::ostream& operator<<(std::ostream&, TypeIn const&);

Unfortunately that extendibility comes at a price as std::stream-based conversions of the basic (int, etc.) types are comparatively slow. To address that boost::lexical_cast implementation has been optimized to avoid std::stream for basic types and Boost.Convert addresses that "optimization" need via pluggable converters. Consequently, if speed is your primary objective and your needs do not go beyond basic types, then deploying Spirit, std::to_string/std::stoi, strtol, boost::lexical_cast (directly or via the Boost.Convert framework) might be an option to explore.

For user-defined types though the boost::cnv::cstream converter seems to consistently outperform boost::lexical_cast in the tests. These results probably reflect different underlying designs.

The standard Boost.Convert deployment pattern is to create a converter or converters once and then re-use them. In turn, boost::lexical_cast also deploys the std::stream engine for user-defined types. However, due to its design boost::lexical_cast creates and then destroys a std::stream instance every time the function is called and the boost::lexical_cast performance table indicates that the "std::stringstream with construction" operation is considerably more expensive compared to "std::stringstream without construction".

Based on the performance data, I tend to conclude that, given type-safety and benefits provided by the Boost.Convert framework, it (with appropriate converters) should probably be the first choice for conversion-related tasks.
