Performance 

boost::cstringstream_converter ccnv; // stringstream-based char converter
int const num_cycles = 1000000;
double p1 = clock();

for (int k = 0; k < num_cycles; ++k)
   boost::lexical_cast<direction>("up");

double p2 = clock();

for (int k = 0; k < num_cycles; ++k)
   boost::convert<direction>::from("up", ccnv);

double p3 = clock();

printf("convert=%.2f, lexical_cast=%.2f\n", (p3 - p2) / CLOCKS_PER_SEC, (p2 - p1) / CLOCKS_PER_SEC);

Performance was tested for a user-defined direction type with the code above. On a Linux box boost::convert performed twice as fast compared to boost::lexical_cast 1.54. Truth be told the test was not fair. Unfortunately, it could not possibly be fair due to different underlying designs.

Due to Boost.Convert design the typical boost::convert usage pattern is to create a converter or converters once and then re-use them (as the code above). Consequently, performance-wise this converter probably takes the fourth (std::stringstream without construction) column of the lexical_cast performance table.

In turn, as boost::convert, for user-defined types boost::lexical_cast also deploys the standard stream framework. However, due to its design boost::lexical_cast creates and then destroys an instance of a stream type during every function call and, consequently, probably takes the third (std::stringstream with construction) much slower column of the lexical_cast performance table.
